关于设计模式那些事

​ 对于设计模式，我的理解是等价于衡量一个开发者的水平标准之一了。对于每个项目的迭代推进都起着至关重要的作用，一份好的代码不仅仅是对用户的友好，也是对开发团队的友好。对于后续的开发效率、bug出现率、系统功能的拓展、标准化、工程化都是一个关键因素。毕竟面向对象=设计模式+算法[数据结构]，所以花时间来输出一下设计模式是非常有必要的。说白了就是怎么搭建更加稳固，牢靠的积木。

# 单例模式

  • 常见单例类：工厂类，配置文件类，日历类。
  • 一个类模板，在系统运行的整个过程中，有且只产生一个实例

# 饿汉式单例

  • 在实例被使用前，提前new一个实例出来。避免并发情况下的线程安全问题
/**
 * @author zhongmaji
 * @description 饿汉式单例
 */
public class HungrySingleton {
    /**
     * 私有构造器
     */
    private HungrySingleton(){}

    private static final HungrySingleton SINGLETON = new HungrySingleton();

    /**
     * 对外暴露获取实例的方法
     * @return HungrySingleton的实例
     */
    public static HungrySingleton getHungrySingleTon(){
        return SINGLETON;
    }
}

# 懒汉式单例

  • 默认加载的时候不初始化，实例在第一次需要的时候，new一个实例
  • 应用：Spring的延迟加载
/**
 * @author zhongmaji
 * @description 懒汉式单例
 */
public class LazySingleton {
    private LazySingleton(){}

    private static LazySingleton lazySingleton;

    public static LazySingleton getLazySingleton(){
        if (lazySingleton == null) {
             //Thread.sleep(0);模拟并发时更好复现问题，让第一个进来的线程放弃CPU时间片，让位给其余线程
            lazySingleton = new LazySingleton();
        }
        return lazySingleton;
    }
}

# 懒汉单例多线程问题

通过上图的测试结果，明确的指出了懒汉式单例的线程安全问题，问题就出在 if (lazySingleton == null)这行代码上，在高并发情况下多个线程同一时刻进行判断，此刻lazySingleton还未实例化，自然在这个时刻的线程全部过检，对LazySingleton这个类进行了多次实例化。那么如何尽可能的避免线程安全问题呢。

  1. 同步方法：给方法加上sychronized关键字，让所有线程串行化执行（同步阻塞）。因此这种方式在高并发的情况下虽然能确保线程安全，但是效率极差!!!
/**
 * @author 众码纪周鸿宇
 * @description 懒汉式单例
 */
public class LazySingleton {
    private LazySingleton(){}
    private static LazySingleton lazySingleton;
    public static synchronized LazySingleton getLazySingleton(){
        if (lazySingleton == null) {
            lazySingleton = new LazySingleton();
        }
        return lazySingleton;
    }
}

结论：双重检查锁单例在一定程度上降低了并发安全性的问题，但不能完全避免。

  • 分析：在JAVA中，JVM会根据处理器的特性（CPU多级缓存，多核处理器等）对机器指令进行重新排序（执行优化），在这个过程中不保证计算机运行的语句和编写的代码顺序一致，也就是指令重排

    • 针对lazySingleton=new LazySingleton();拆分

      • 首先虚拟机会为实例分配一定的内存空间
      • 开始初始化对象
      • 将初始化的对象地址指向分配的内存空间

    • 编译器为了性能原因，可能会将第二步和第三步进行重排序

    • 接下来分析问题发生场景

Time(执行时间) Thread-[name：A] Thread02-[name：B]
T1 A检测lazySingleton为空 未入方法栈
T2 A获取到锁 进入方法栈
T3 再次检测到lazySingleton为空 外层判断是否为空中
为lazySingleton分配内存空间 外层判断是否为空中
T5 将lazySingleton指向内存空间
（此处已经发生指令重排）		 检测到lazySingleton不为空，返回实例
T6 已完成地址指向，刚准备初始化 访问实例
T7 初始化lazySingleton

  • T6时刻B线程到的lazySingleton仅仅是一个初始化未完成初始化的对象，在这里就可能出现不可控制的问题

  • 解决方案：

    • 同步方法是一种（不推荐）
    • 使用volatile关键字：关于如何保证数据的可见性、防止指令间的重排序，可参考计算机内存模型（cpu高速缓存[三层L1~L3]，主内存，缓存一致性协议MESI），以及java内存模型JMM。这里不过多阐述哈（不是本文中心点），后续并发编程中补充。
    • 实际上作者本人查询了关于volatile能不能解决懒汉式单例线程安全问题，学术上还存在一定争议哈，所以请读者持保留态度。
/**
 * @author 众码纪周鸿宇
 * @description 懒汉式单例
 */
public class LazySingleton {
  private LazySingleton(){}
  private volatile static Lazysingleton lazySingleton;
  
  public static LazySingleton getLazySingleton(){
    if(lazySingleton==null){
      sychronized(Singleton.class){
        if(lazySingleton==null){
          lazySingleton = new LazySingleton();
        }
      }
    }
    return lazySingleton;
  }
}
